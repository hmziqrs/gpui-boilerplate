# Native Local Notifications Plan

## Goal

Add native OS-level local notifications to this GPUI app with:

- a proper macOS permission flow,
- a capability-aware backend shape for macOS, Linux, and Windows,
- and explicit degraded-mode behavior when the preferred native path is unavailable.

This plan is only about local OS notifications triggered by the running app.

## Out Of Scope

- remote push notifications
- server-driven push delivery
- APNs / FCM / WNS integration
- replacing the app's existing in-window GPUI toast system
- a cross-platform scheduling abstraction for delayed notifications

Push notifications are a different system and need a separate plan.

## Terminology

- In-app notification:
  - current GPUI UI toast shown with `window.push_notification(...)`
  - only visible inside the app window
- Local OS notification:
  - native desktop notification generated by this app on the current machine
  - the target of this plan
- Push notification:
  - remote notification delivered through a vendor/system push channel
  - explicitly out of scope here

## Current State

- The app currently shows only in-window GPUI toasts through `window.push_notification(...)`.
- Current call sites:
  - `src/views/settings.rs`
  - `src/views/form_page.rs`
  - `src/title_bar.rs`
- Good integration seams already exist:
  - `src/app.rs` for global state and startup wiring
  - `src/main.rs` for module registration
  - `src/tray.rs` for macOS-specific runtime behavior
- The current local build output is a raw binary at `target/debug/gpui-starter`.
- There is no checked-in `.app` bundle path, `Info.plist`, or signing workflow yet.

## Audit Findings

### 1. The prior plan blurred three separate systems

These must stay distinct:

- in-app GPUI toasts
- native local OS notifications
- remote push notifications

We should not present native local notifications as a stepping stone to push. They share product language, but not delivery architecture.

### 2. Cross-platform permission support is uneven

`user-notify` only exposes meaningful permission APIs on macOS.

Implications:

- macOS:
  - real permission lookup and first-time request path exist
- Linux:
  - no matching permission API in `user-notify`
- Windows:
  - `user-notify` treats permission as not needed / no-op

The app must be capability-driven and must not pretend Linux or Windows have the same permission model as macOS.

### 3. Delayed scheduling was overstated in the prior plan

The shared request model must not include delayed delivery in v1.

Reason:

- `user-notify` exposes immediate send, not a cross-platform scheduling abstraction
- `notify-rust` scheduling exists only on macOS
- a generic `deliver_after` field in the shared API would make the abstraction lie

### 4. GPUI toast is not an equivalent native fallback

A GPUI toast is only visible when the app window is active enough for the user to see it.

That means:

- it is acceptable as foreground feedback
- it is not equivalent to a background OS notification
- when native delivery fails for a background-worthy event, the app is in a degraded state, not a successful fallback state

### 5. macOS runtime requirements are real

`user-notify`'s macOS path requires a bundled app with a bundle identifier, and its docs call out signing/runtime constraints.

`notify-rust` is a useful degraded backend, but it is not a guaranteed rescue path for every unbundled macOS dev run.

### 6. License remains a gate

`user-notify` is `LGPL-3.0-or-later`.

This is not a footnote. It is an adoption gate that should be confirmed before implementation lands.

## Technical Strategy

### Primary backend

Use `user-notify` as the primary backend for immediate native local notifications.

Why:

- it is explicitly cross-platform for macOS, Linux, and Windows
- on macOS it uses the modern `UNUserNotificationCenter` path
- it exposes the macOS permission APIs we actually need
- it supports richer notification features than simpler delivery-only crates

Limits we must respect:

- permission APIs are meaningfully implemented only on macOS
- its macOS runtime requirements are stricter than a raw `cargo run`
- scheduling should not be assumed as part of the shared abstraction

### Secondary backend

Use `notify-rust` as the explicit degraded delivery backend.

Why:

- it is mature and established
- it gives us a separate native delivery path when `user-notify` is unavailable
- it is a good fit for best-effort summary/body delivery

Limits we must respect:

- it is not the source of truth for macOS permission state
- its macOS path is less capable than the `UNUserNotificationCenter` path
- it does not fix the conceptual gap between native delivery and in-app toasts

### Final degraded mode

Keep GPUI in-window toasts as foreground-only user feedback.

That gives us this practical order:

1. `user-notify`
2. `notify-rust`
3. visible in-app feedback plus degraded-state reporting

Important distinction:

- step 3 is not equivalent success for background-worthy events
- step 3 is only acceptable for foreground-visible events

## Platform Scope

The architecture should be cross-platform from the start, but capability tiers must be explicit.

### Tier A: v1 required

- immediate native local notification delivery when a native backend works
- backend selection and fallback
- clear degraded-state reporting

### Tier B: macOS-only in v1

- permission state lookup
- first-time permission request
- settings recovery guidance

### Tier C: later, platform-specific

- richer actions / interactivity
- notification history integration
- tray/badge recovery for degraded background-worthy events

### Tier D: separate later phase

- scheduled local notifications

### Tier E: separate future plan

- push notifications

## Permission Model

### Public state model

Use a capability-aware permission enum:

- `Unknown`
- `Unsupported`
- `Unavailable`
- `NotDetermined`
- `Denied`
- `Authorized`

Semantics:

- `NotDetermined`, `Denied`, and `Authorized` should only come from a real platform-backed signal
- `Unsupported` means the active backend/platform does not offer a meaningful permission API
- `Unavailable` means the platform could support it, but this runtime/build cannot currently provide it

### macOS

Behavior:

- read permission state only through `user-notify`
- request permission only through `user-notify`
- if denied, do not re-prompt
- show UI guidance and an `Open Settings` recovery path
- if the app is not bundled/signed correctly, treat permission management as `Unavailable`

### Linux

Behavior:

- attempt native delivery
- do not map `user-notify`'s non-macOS no-op behavior to `Authorized`
- report permission state as `Unsupported`

### Windows

Behavior:

- attempt native delivery
- do not map `user-notify`'s non-macOS no-op behavior to `Authorized`
- report permission state as `Unsupported` unless we later add a real Windows-specific status path

## Delivery Model

### Request model for v1

Suggested shared request:

- `title`
- `body`
- `play_sound`
- `thread_id`
- `category`
- `prefer_native`
- `importance`

Suggested supporting enums:

- `NotificationImportance`
  - `ForegroundOnly`
  - `BackgroundWorthy`

Do not include `deliver_after` in v1.

### Why `importance` matters

This field decides whether in-app feedback can count as an acceptable degraded outcome.

Rules:

- `ForegroundOnly`:
  - GPUI toast fallback is acceptable
- `BackgroundWorthy`:
  - GPUI toast does not satisfy delivery semantics by itself
  - failures must be surfaced as degraded state

## Proposed Architecture

### Module layout

- `src/notifications/mod.rs`
  - public API
  - shared types
- `src/notifications/service.rs`
  - backend selection
  - send-time fallback logic
  - degraded-state policy
- `src/notifications/backend/mod.rs`
  - backend trait + wiring
- `src/notifications/backend/user_notify.rs`
  - `user-notify` adapter
- `src/notifications/backend/notify_rust.rs`
  - `notify-rust` adapter
- `src/notifications/backend/ui_only.rs`
  - explicit no-native backend for visible in-app fallback only

### Core types

Suggested types:

- `NotificationPermissionState`
- `NotificationBackendKind`
  - `UserNotify`
  - `NotifyRust`
  - `UiOnly`
- `NotificationImportance`
  - `ForegroundOnly`
  - `BackgroundWorthy`
- `NotificationRequest`
- `NotificationSendResult`
  - `backend_used`
  - `degraded`
  - `delivered_natively`
  - `error_summary`
- `NotificationCapabilities`
  - `can_request_permission`
  - `can_read_permission_state`
  - `can_send_immediate_native`
  - `can_send_interactive`
  - `requires_packaged_runtime`

### Backend trait

Define a trait around the features we actually need:

- `backend_kind()`
- `capabilities()`
- `refresh_permission_state()`
- `request_permission()`
- `send()`
- `open_system_settings()`

Important rules:

- `refresh_permission_state()` may legitimately return `Unsupported`
- `request_permission()` must not be exposed unless the backend can actually do it
- `send()` should degrade rich requests to simple summary/body delivery before failing over

### Global state

In `src/app.rs`, register global notification state such as:

- `NativeNotificationState`
  - `enabled_by_user: bool`
  - `permission: NotificationPermissionState`
  - `active_backend: NotificationBackendKind`
  - `last_backend_error: Option<SharedString>`
  - `degraded_reason: Option<SharedString>`

If we later need stronger recovery for background-worthy failures, extend this with retained notification history or unread event state.

## Backend Strategy

### Startup order

1. try to initialize `user-notify`
2. if that fails, initialize `notify-rust`
3. if that fails, initialize `UiOnly`

### Send order

1. attempt send on the selected backend
2. if it fails, attempt the next native backend
3. if all native paths fail:
   - for `ForegroundOnly`, show in-app feedback
   - for `BackgroundWorthy`, mark the result as degraded and preserve visible failure state

### `user-notify` -> `notify-rust`

Use this fallback when:

- `user-notify` manager initialization fails
- macOS runtime constraints are missing
- a richer request can be degraded to a simple native notification

Do not use this fallback for:

- macOS permission lookup
- macOS permission requests

### `notify-rust` -> `UiOnly`

Use this fallback when:

- the OS notification daemon or API path is unavailable
- the send call errors
- the user has disabled native notifications in app settings

Important limitation:

- `UiOnly` is acceptable only for `ForegroundOnly` notifications
- `BackgroundWorthy` failures must remain visible as degraded failures, not silent success

## File-By-File Implementation Plan

### 1. Add dependencies

Files:

- `Cargo.toml`

Work:

- add direct `user-notify`
- add direct `notify-rust`

Use both as direct dependencies even though `user-notify` may depend on `notify-rust` internally on some paths. We want an explicit app-owned fallback policy.

### 2. Add the notification service and backend adapters

Files:

- `src/main.rs`
- `src/notifications/mod.rs`
- `src/notifications/service.rs`
- `src/notifications/backend/mod.rs`
- `src/notifications/backend/user_notify.rs`
- `src/notifications/backend/notify_rust.rs`
- `src/notifications/backend/ui_only.rs`

Work:

- add the `notifications` module
- define the shared types and backend trait
- implement a `user-notify` adapter
- implement a `notify-rust` adapter
- implement a `UiOnly` adapter
- implement startup backend selection
- implement send-time fallback chaining
- implement degraded-result reporting

### 3. Initialize state at startup

Files:

- `src/app.rs`

Work:

- create the notification service during `app::init`
- store active backend and permission state in app-global state
- on startup:
  - try `user-notify`
  - otherwise fall back to `notify-rust`
  - otherwise mark `UiOnly`
- refresh permission state asynchronously only when the selected backend supports it

### 4. Build the Settings UI for local notifications

Files:

- `src/views/settings.rs`
- `i18n/en/gpui-starter.ftl`
- `i18n/zh-CN/gpui-starter.ftl`

Work:

- replace the demo notification button with a proper settings section
- show backend and permission status
- gate permission actions by backend capability
- add a `Test native notification` action that always goes through the notification service
- explain degraded mode clearly
- explain that GPUI in-app toasts remain a separate feature

### 5. Route only selected call sites through the service

Files:

- `src/views/settings.rs`
- `src/views/form_page.rs`
- `src/title_bar.rs`

Work:

- do not treat all current `window.push_notification(...)` call sites as native notification candidates
- preserve the existing GPUI toast system as a separate subsystem
- route only call sites whose product semantics justify OS-level delivery

Suggested first pass:

- Settings `Test notification`: always service-based
- form submit success: keep in-app by default unless there is a real background alert use case
- title-bar bell: keep as GPUI toast until the app has a real notification feed

### 6. Add macOS bundle/signing support for the primary backend

Files:

- `build.rs` or `scripts/macos-dev-app.sh`
- optional bundled metadata file

Work:

- build a minimal `.app` bundle for local testing
- set a stable bundle identifier
- add ad-hoc signing or a documented signing path for development

This is required because `user-notify`'s macOS path expects a real app package and bundle identifier.

### 7. Implement degraded-state policy instead of pretending full fallback success

Files:

- `src/notifications/service.rs`
- `src/notifications/backend/user_notify.rs`
- `src/notifications/backend/notify_rust.rs`

Work:

- if permission APIs are unavailable, report `Unsupported` or `Unavailable` instead of guessing
- if a request includes unsupported rich features, degrade it to a simple notification before failing over
- persist the last backend failure so Settings can explain the current state
- do not mark a `BackgroundWorthy` request as successful just because an invisible in-app fallback was attempted

## Verification Plan

### Build verification

- `cargo check`
- `cargo build`

Confirm:

- both crates compile in this workspace
- backend selection compiles on macOS
- Linux and Windows paths remain guarded appropriately if cross-compiling is deferred
- the implementation does not pretend unsupported permission APIs exist on Linux or Windows

### Manual verification on macOS

1. launch from the bundled `.app`
2. confirm active backend is `user-notify`
3. confirm permission state is readable
4. request permission
5. accept permission and trigger a test notification
6. deny permission and relaunch
7. confirm the app:
   - reads `Denied`
   - does not re-prompt
   - offers recovery UI
   - clearly reports degraded state if native delivery is unavailable

### Manual fallback verification on macOS

1. launch a dev build that lacks the proper runtime shape
2. confirm backend degrades away from `user-notify`
3. trigger a test notification
4. confirm:
   - fallback backend selection is visible
   - send either works through `notify-rust` or lands in `UiOnly`
   - the app never lies about permission state

### Manual verification on Linux

1. trigger a simple native notification
2. confirm it uses the native backend when a notification daemon exists
3. kill or disable the notification daemon if practical
4. confirm send failure degrades to visible in-app feedback only for `ForegroundOnly`
5. confirm `BackgroundWorthy` sends report degraded delivery rather than fake success

### Manual verification on Windows

1. trigger a simple native notification
2. confirm it uses the native backend when toast delivery is available
3. confirm native failure degrades to visible in-app feedback only for `ForegroundOnly`
4. confirm `BackgroundWorthy` sends report degraded delivery rather than fake success

## Acceptance Criteria

- the app has a single native local notification service API that views call into when OS-level delivery is intended
- `user-notify` is the preferred backend when available
- `notify-rust` is used as an explicit degraded backend
- GPUI in-app toasts remain a separate subsystem
- macOS permission state and permission requests are handled only through the primary backend
- Linux and Windows do not report fake permission states
- the Settings page shows:
  - current backend
  - permission state
  - degraded/fallback status
- the plan clearly separates:
  - native local notifications
  - in-app notifications
  - push notifications
- background-worthy notification failures are surfaced as degraded delivery, not as silent success

## Risks And Open Questions

### Risk: `user-notify` license

If `LGPL-3.0-or-later` is unacceptable, we should stop before implementation and replace the primary backend strategy.

### Risk: macOS dev-mode reliability

`notify-rust` on macOS is a degraded path, not a guaranteed escape hatch.

That is why the Settings page must expose selected backend and degraded mode clearly.

### Risk: duplicate delivery

If we send both a native banner and a GPUI toast for the same foreground event, the UX will become noisy.

Default policy should be:

- background-worthy events -> native local notification
- immediate foreground feedback -> GPUI toast

## Suggested Implementation Order

1. confirm the `user-notify` license is acceptable
2. add both dependencies and the service module
3. implement the `user-notify` adapter
4. implement the `notify-rust` adapter
5. implement the `UiOnly` adapter and degraded-state policy
6. wire the Settings UI
7. add the macOS bundled dev path
8. route the first real native-notification use case through the service
9. write a separate push-notifications plan later
